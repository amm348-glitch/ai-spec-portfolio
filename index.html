Methodology

I use a spec-driven workflow when building with AI. Requirements are written first, then used to guide AI-assisted implementation in bounded sprints. Each sprint is verified against acceptance criteria before moving forward. This reduces drift and makes AI output predictable and auditable.

Featured Project — AI Spec-Driven Portfolio

The Brief & Constraints

The goal of this project was to build a personal portfolio site that demonstrates a spec-driven development workflow using AI. The emphasis was on process clarity rather than visual polish.

  • Static site (HTML + CSS only)
  • Hosted on GitHub Pages
  • Short timeline
  • Focus on documentation and reproducibility

The Spec

Before implementation, a written specification was created to lock the site structure, design direction, and acceptance criteria. This spec acted as the source of truth for all development decisions.

View the locked implementation spec

AI Orchestration

AI was used as an implementation assistant rather than a source of final answers. Early exploration was used to evaluate design directions and content structure. Once decisions were made, the spec was locked to prevent scope drift.

Development was executed in bounded sprints with explicit goals and acceptance criteria. AI-assisted output was iterated on only when it failed to meet the written spec.

The Result

  • Live deployed site (GitHub Pages)
  • GitHub repository

Sprint 001 resulted in a fully deployed site scaffold with documented planning, a locked spec, and a live production URL.

Reflection

  • Writing the spec first significantly reduced ambiguity during implementation.
  • Locking decisions early prevented unnecessary visual and structural churn.
  • Future iterations could expand the case study depth and add automated checks.
